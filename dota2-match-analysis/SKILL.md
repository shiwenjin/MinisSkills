# Dota2 比赛分析技能

基于单场 Dota2 对局或单个玩家账户数据生成可靠、可读的分析。先获取数据，再提炼结果、阵容、玩家表现、关键事件、比赛走势或玩家画像；不要把输出写成原始字段转抄。

优先使用 `scripts/fetch_match.py` 处理比赛、使用 `scripts/fetch_player.py` 处理账户数据并生成基础归一化结果；当用户需要更深入的分析口径时，再读取 `references/analysis-guidelines.md`。英雄中英文名称和常用别称优先参考 `references/hero-name-aliases-with-id.csv`。

## 必要输入

- `match_id`：比赛 ID
- `account_id`：玩家账户 ID

根据用户意图选择输入：

- 分析单场比赛时需要 `match_id`
- 分析某位玩家近期表现、常用英雄、胜率时需要 `account_id`
- 分析“最近一场比赛”时，先需要 `account_id`，再从玩家近期比赛中取最近一场的 `match_id`

如果缺少 `match_id`，直接向用户索取，不要猜测。

如果缺少 `account_id`，按这个顺序处理：

1. 先检查用户记忆里是否已有可用的 `account_id`
2. 只有在记忆中找不到时，再请用户输入 `account_id`

不要猜测 `account_id`，也不要把其他数字误当成玩家账户 ID。

如果用户问的是“最近一场比赛情况怎么样”“我上一把打得怎么样”这类最近对局问题，按这个顺序执行：

1. 先获取 `account_id`；如果用户没给，就先查用户记忆，记忆里没有再向用户索取
2. 使用 `scripts/fetch_player.py {account_id}` 获取玩家数据
3. 从玩家近期比赛中提取最近一场的 `match_id`
4. 再使用 `scripts/fetch_match.py {match_id}` 获取该场比赛详情
5. 最后按比赛分析报告输出，不要只给玩家概况

## 执行流程

### 1. 获取比赛或玩家数据

#### 分析比赛 (`match_id`)
```bash
python3 scripts/fetch_match.py {match_id}
```
脚本会自动处理缓存、补充英雄中文名（基于 `references/hero-name-aliases-with-id.csv`）并输出归一化 JSON。

#### 分析玩家 (`account_id`)
```bash
python3 scripts/fetch_player.py {account_id}
```

#### 分析“最近一场比赛”
如果用户未提供 `match_id` 但指定了 `account_id`（或通过记忆/询问获取），先运行玩家脚本，从 `recentMatches` 中提取最近一条的 `match_id`，再运行比赛脚本。

**注意**：脚本已内置英雄名称映射。如果结果中偶有中文名缺失，请参考 `references/hero-name-aliases-with-id.csv` 手动修正。

### 2. 比赛分析要点

确认数据完整性（`players`, `radiant_win`, `duration` 等）后，分析以下维度：

- **概况**：胜负、时长、总人头比、比赛模式/段位。
- **阵容**：基于 `players` 区分阵营（slot 0-4 天辉, 128-132 夜魇）。评估控制、爆发、推塔、肉山及后期能力。
- **表现**：综合 KDA、伤害（英雄/建筑）、参战率、经济（GPM/XPM/NetWorth）和辅助数据（治疗/控制）。评选 MVP 时不要只看 KDA，要结合对局势的实际贡献。
- **转折**：基于 `objectives` (一血/推塔/肉山) 和 `teamfights` (团战) 梳理关键节点。找出决定比赛走势的那几波团战或买活。
- **聊天**：如有 `chat`，提取反映局势心态的关键对话，简述氛围（互喷、整活、战术交流）。

**重要**：输出前必须根据 `references/hero-name-aliases-with-id.csv` 核对所有英雄名称，确保使用的是准确的中文官方名称或玩家通用的中文别称（如“敌法”、“小黑”）。

## 输出要求

默认输出为精简版结构化 Markdown，优先把高信息量内容压进 4 到 6 个小节，单个小节控制在 2 到 5 行；只有用户明确要求详细复盘、逐人分析、完整时间线时，才展开成长版。

比赛分析默认使用以下短版结构：

```markdown
# Dota2 比赛分析报告

## 一句话结论
- 谁赢了
- 怎么赢的
- 谁是最关键的人

## 关键信息
- 比赛时长、比分、模式、区域
- 双方阵容一句话判断

## 胜负手与关键转折
- 1 到 3 个最关键节点
- 为什么这些节点决定走势

## 玩家亮点
- 只重点写 2 到 4 人
- 每人 1 行结论 + 1 个核心证据

## 补充信息
- 团战、聊天、对线、阵容克制里只保留最有解释力的内容
- 没有价值的模块直接省略
```

如果用户提供的是 `account_id`，默认输出为精简版玩家账户分析报告，建议保持以下顺序：

```markdown
# Dota2 玩家账户分析报告

## 核心判断
- 当前状态如何
- 强项英雄或打法是什么
- 一句话结论

## 近期表现
- 最近样本、胜率、简单走势

## 常用英雄
- 只列前 3 到 5 个最有代表性的英雄
- 每个英雄只保留场次、胜率、简短判断

## 风格与建议
- 用 KDA / 场均数据概括打法
- 总结 1 到 2 条最实际的建议
```

## 写作要求

- 先下结论，再给证据
- 突出"为什么赢、为什么输、谁打得最好"
- 避免把输出写成 API 字段清单
- 数据不足时明确说明，不要脑补
- 除非用户要求详细版，否则默认控制篇幅，优先保留高信息量内容
- 默认不要逐个玩家平均着墨，重点信息前置，次要模块能合并就合并
- 能用一句话说清的，不要扩成一段；能合并成同一节的，不要拆很多标题
- 默认不输出冗长时间线、全员点评、整段聊天记录、重复数据解释
- 比赛报告默认使用毒舌评论风格，像解说搭档在直播中互相吐槽那种感觉
- 毒舌要有梗有料，基于真实数据和比赛走势，不要为了毒而硬喷
- 重点diss关键失误、迷之操作、犯罪现场，赢的那方也可以酸两句"赢得太轻松没意思"
- 如果有聊天记录，优先把聊天摘录和比赛走势联系起来，写出一点"场上在打，场下也在斗嘴"的感觉
- 可以适度使用扎心小标题或一句话吐槽，但不要变成纯玩梗文

## 错误处理

- `match_id` 无效：提示用户检查比赛 ID
- `account_id` 无效：提示用户检查玩家账户 ID
- API 失败：说明获取数据失败，并保留可重试建议
- 数据不完整：照常输出可分析部分，并标记缺失模块
